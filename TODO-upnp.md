UPnP / DLNA handling
====================

This proxy is already advertising itself as a UPnP `MediaRenderer` via SSDP and `description.xml` in `avr_discovery.py`, but it does **not** implement the standard UPnP service endpoints like:

- `/RenderingControl/desc.xml`
- `/ConnectionManager/desc.xml`
- `/AVTransport/desc.xml`

When other devices (TVs, OS media frameworks, etc.) discover the proxy, they try to fetch these service descriptions and we currently log warnings such as:

> Client X.X.X.X request: GET /RenderingControl/desc.xml -> no handler

This TODO tracks adding at least minimal handling for these requests so the proxy behaves more like a real UPnP renderer.

References
----------

- `denon_proxy.py`
  - Overall proxy lifecycle and configuration (`enable_ssdp`, `ssdp_http_port`, `avr_host`, etc.).
  - Awareness that `denonavr` is used for HTTP/UPnP-ish state sync.
- `avr_discovery.py`
  - SSDP advertising (`SSDPProtocol`, `ssdp_response`, `run_discovery_servers`).
  - UPnP device description XML (`description_xml`) that already declares `MediaRenderer:1`.
  - HTTP device description handler (`DeviceDescriptionHandler`) that currently serves:
    - `/description.xml`
    - `/goform/Deviceinfo.xml`
    - `/goform/AppCommand.xml` (POST)
    - `MainZoneXml` variants
  - This is also where we currently log the “-> no handler” warning for these UPnP paths.
- `avr_state.py`
  - Source of power / volume / input / mute / sound mode state that we could expose via UPnP services.

Open questions
--------------

- **Forward vs. emulate?**
  - Can/should we simply **forward UPnP control requests unmodified** to the physical AVR (if `avr_host` is configured)?
  - Or is it better/safer to **emulate a minimal subset** (e.g. just enough to keep HA / common clients happy) and *not* proxy arbitrary SOAP actions?
  - If forwarding, do we:
    - Use `httpx` (already optional in `avr_discovery.py`) to send the same request to the real AVR and relay the response?
    - Need to rewrite host/port/URLs in the response XML so follow‑up URLs point back to the proxy instead of the AVR?
- **Multi‑client safety**
  - Do Home Assistant / `denonavr` or other UPnP clients care if **multiple control points** talk to the same physical device concurrently via UPnP while others use telnet via the proxy?
  - Are there known issues with allowing **both**:
    - HA (or others) using the proxy as a UPnP renderer, and
    - Direct `denonavr` HTTP/UPnP traffic to the physical AVR (if users also have that configured separately)?
- **Minimum viable UPnP feature set**
  - What is the smallest set of services/actions we must support to:
    - Stop the “no handler” warnings.
    - Avoid confusing typical control points (e.g. they don’t immediately give up or spam retries).
  - Do we need **full**:
    - `RenderingControl` (volume, mute, picture)?
    - `ConnectionManager` (protocol info, connection IDs)?
    - `AVTransport` (play/pause/stop/seek, track metadata)?
  - Or can we respond with valid XML that advertises a very limited, mostly‑no‑op implementation for anything we don’t actually support?

Implementation ideas / steps
----------------------------

1. **Add UPnP service description handling to `DeviceDescriptionHandler` in `avr_discovery.py`.**
   - Extend the GET routing in `DeviceDescriptionHandler.data_received` to recognize:
     - `/RenderingControl/desc.xml`
     - `/ConnectionManager/desc.xml`
     - `/AVTransport/desc.xml`
   - For now, serve **static XML templates** embedded in `avr_discovery.py` or generated by helper functions, describing:
     - `urn:schemas-upnp-org:service:RenderingControl:1`
     - `urn:schemas-upnp-org:service:ConnectionManager:1`
     - `urn:schemas-upnp-org:service:AVTransport:1`
   - Make sure the **controlURL** and **eventSubURL** in these templates point back to the proxy (e.g. `/upnp/control/RenderingControl` etc.) so future work can hook them.

2. **Decide on control URL strategy.**
   - Define canonical local paths for control and event URLs, e.g.:
     - `/upnp/control/RenderingControl`
     - `/upnp/control/ConnectionManager`
     - `/upnp/control/AVTransport`
     - `/upnp/event/RenderingControl`
     - `/upnp/event/ConnectionManager`
     - `/upnp/event/AVTransport`
   - Update the new `desc.xml` templates to point to those URLs.
   - Extend `DeviceDescriptionHandler` to:
     - Parse SOAPAction headers for POSTs to these control URLs.
     - For now, respond with valid but mostly stub SOAP envelopes (e.g. success with minimal state) rather than implementing full behavior.

3. **Explore simple pass‑through mode to the physical AVR (optional).**
   - If `avr_host` is set and `httpx` is available:
     - Try a naïve pass‑through implementation:
       - For each incoming UPnP control POST (e.g. `AVTransport`), forward the request verbatim to the AVR at:
         - `http://{avr_host}:{port}/{same-path}` where `port` is likely 8080 or 60006 (we already probe these in `_fetch_avr_description`).
       - Relay status code, headers (sanitized), and body back to the client.
     - Edge questions:
       - Does the physical AVR expect slightly different paths (e.g. `/upnp/control/AVTransport1` vs `/AVTransport/Control`)?
       - Do responses embed absolute URLs/UDNs that must be rewritten to refer to the proxy?
   - Add a config toggle such as:
     - `upnp_passthrough: true|false`
   - When disabled or when `avr_host` is empty (virtual AVR), fall back to the stub implementation from step 2.

4. **Integrate UPnP actions with `AVRState` and telnet commands (longer‑term).**
   - Map key UPnP actions to the existing telnet / state machinery:
     - `RenderingControl.SetVolume` -> send `MVxx` via `AVRConnection` and update `AVRState.volume`.
     - `RenderingControl.SetMute` -> send `MUON` / `MUOFF`.
     - `AVTransport.Play` / `Pause` / `Stop` -> if feasible, decide what these mean for a Denon AVR (often they map to input/network control rather than telnet commands; may not be worth fully supporting).
   - Reuse code paths in `denon_proxy.py` that already:
     - Apply optimistic state updates.
     - Broadcast new state to telnet clients and web UI.
   - Ensure we do **not** introduce feedback loops where:
     - UPnP actions change AVR state, which triggers telnet responses, which we then re‑interpret as further UPnP events.

5. **Consider eventing support (UPnP subscriptions).**
   - Many control points ignore events and simply poll, but some may subscribe.
   - Decide whether to:
     - Implement minimal event subscription endpoints (respond with 200, but maybe do not push notifications initially), or
     - Explicitly reject subscription attempts with a clear error.
   - If implementing events:
     - Reuse the existing state change notifications (`_notify_web_state`, `AVRState`) to build and send UPnP event messages.

6. **Compatibility and safety checks.**
   - Test behavior with:
     - Home Assistant’s `denonavr` integration using the proxy’s SSDP / UPnP endpoints only.
     - A generic DLNA control point (e.g. VLC, Windows “Play To”, or a phone app) to see what they expect from:
       - `RenderingControl`
       - `ConnectionManager`
       - `AVTransport`
   - Observe:
     - Whether these clients behave correctly when:
       - The proxy is only partially implementing services.
       - Another client is talking directly to the physical AVR.
   - Update this TODO with notes:
     - Do common `denonavr` clients care if **another** UPnP control point is active?
     - Are there any race conditions or state inconsistencies between direct HTTP/UPnP use of the AVR and proxy‑mediated telnet control?

7. **Configuration surface and docs.**
   - Add new config options (sketch):
     - `enable_upnp_services` (default `True` when `enable_ssdp` is `True`).
     - `upnp_passthrough` (default `False` for safety).
   - Document in `config.sample.yaml` and project README:
     - What UPnP features the proxy provides.
     - Trade‑offs of pass‑through vs. emulated mode.
     - Any known limitations when multiple clients use UPnP and telnet simultaneously.
